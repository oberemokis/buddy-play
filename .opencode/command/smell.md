Проведи ревью изменённых файлов на code smells по правилам проекта.

## Какие файлы проверять

ТОЛЬКО изменённые: список ниже в секции «ИЗМЕНЁННЫЕ ФАЙЛЫ». Проверяй файлы с кодом
(`.ts`, `.vue`, `.tsx`); конфиги, локфайлы, markdown и сгенерированные файлы
(`*.d.ts`, `tsconfig.tsbuildinfo`) пропускай. Если список пуст — так и скажи, ничего не выдумывай.

## По каким правилам

Единственный источник правды — файл `.rules` в корне проекта. Прочитай его целиком
и проверяй КАЖДЫЙ изменённый файл по КАЖДОЙ применимой секции: именование
(префиксы/суффиксы), функции, импорты, TypeScript, моделирование состояний,
null-политика, Effect-TS, Vue, Pinia, модули, комментарии, управление потоком,
константы, обработка ошибок, дублирование, тесты.

Не придумывай собственных правил сверх `.rules`. Если сомневаешься, нарушение это
или норма, — помечай как «спорно», а не как нарушение.

## Формат отчёта

Группируй по файлам. Для каждой находки:

```
apps/web/posts/src/components/PostCard.vue:42
  Правило: TypeScript → никакого `any`
  Найдено: `const data: any = await response.json()`
  Как исправить: типизировать через Effect Schema (`PostSchema`)
```

В конце — сводка: сколько файлов проверено, сколько нарушений, топ-3 самых частых правила.
Если нарушений нет — короткое «чисто, N файлов проверено».

## Что НЕ делать

- НЕ исправляй код — только отчёт. Исправления пользователь запросит отдельно.
- НЕ проверяй файлы вне списка изменённых.
- НЕ дублируй ошибки компилятора (`tsc` сам их найдёт) — только стиль и архитектура из `.rules`.
